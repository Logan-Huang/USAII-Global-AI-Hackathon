# Android Migration Blueprint — AI-Powered Asylum Aid

How the existing web app becomes a **fully native Android app** (Kotlin + Jetpack Compose)
while reusing the same Express backend, the same API contract, and the same responsible-AI
guardrails. This mirrors what `ios/` did for SwiftUI.

---

## 1. What the current app is, and where the seams are

The product is a thin browser client over a local-proxy backend:

```
public/ (browser)  ──HTTP──▶  server/ (Express)  ──HTTPS──▶  Anthropic Claude
                                     │                         (key server-side only)
                                     └──▶ Open-Meteo (geocode) + OSM Overpass (places)
```

The backend is **stateless, storage-free, key-holding**. The browser holds all chat state
in memory and re-sends history on every turn. That design is exactly what makes a second
native client cheap: **the API is already the contract**, and there is no shared session
state to migrate.

### 1a. The frontend (what we re-implement natively)

`public/` is vanilla JS with no build step. The pieces that carry *behavior* (and therefore
must be re-implemented, not just restyled) are:

| Web source | Responsibility | Android home |
|---|---|---|
| `index.html` + `styles.css` | Screens, brand palette | Compose screens + `ui/theme/Theme.kt` |
| `app.js` → NDJSON reader | Read `response.body` line-by-line, render deltas | `net/ChatStream.kt` (OkHttp + `BufferedSource.readUtf8Line`) |
| `app.js` → `renderMarkdown` | Safe markdown (escape→tokenize, links http-only, lists keep numbering across blank lines) | `md/MarkdownRenderer.kt` (pure) + `ui/MessageBubble.kt` (→ `AnnotatedString`) |
| `app.js` → country dropdown (`Intl.DisplayNames`) | Localized country names | `i18n/Countries.kt` (`java.util.Locale.getDisplayCountry`) |
| `app.js` → Leaflet map | "Help near you" map | `ui/MapScreen.kt` (osmdroid — same OSM tiles, no API key) |
| `app.js` → resources modal | "Find legal help" directory | `ui/ResourcesScreen.kt` |
| `i18n.js` (`t`, `detectLanguage`, `isRTL`, `buildInitialMessage`) | i18n + first-message composition | `i18n/LocalizationStore.kt` (a direct port) |
| `i18n.js` + `ui-translations.js` + `languages.js` (100 langs) | Translation tables | Bundled JSON assets, generated by `scripts/gen-android-strings.js` |

### 1b. The API routes (what we reuse unchanged)

No backend change is required. The four endpoints in `server/server.js` are the contract:

| Endpoint | Method | Native need |
|---|---|---|
| `/api/chat` | POST | NDJSON stream `{type:"delta"|"done"|"error"}` |
| `/api/resources?country=` | GET | JSON `{official,legalAid,global}` |
| `/api/geocode?q=` | GET | JSON `{lat,lon,displayName}` |
| `/api/places?lat&lon&radius` | GET | JSON `{count,places[]}` |

The only browser-specific concern in the backend is **CORS / CSP**, and it does not apply to
a native client: `URLConnection`/OkHttp are not subject to same-origin policy, and the CSP
header is simply ignored off-browser. **So the server needs zero changes for Android.**

## 2. What must be *created* to serve a native mobile client

Nothing on the server. Everything is on the client, and it falls into four buckets:

1. **A transport layer that speaks the existing contract.**
   - JSON GETs → `net/ApiClient.kt` (OkHttp + kotlinx.serialization).
   - The streaming POST → `net/ChatStream.kt`. The web client reads `response.body` with a
     `TextDecoder` and splits on `\n`; the native equivalent is OkHttp's streaming
     `ResponseBody.source()` + `readUtf8Line()`, surfaced as a Kotlin `Flow<String>` of deltas.

2. **A faithful re-implementation of the browser-only logic** that has no server endpoint:
   safe markdown rendering, the i18n fallback chain, RTL detection, country-name localization,
   and `buildInitialMessage`. These are ported verbatim from `app.js`/`i18n.js` so behavior
   matches byte-for-byte where it matters (e.g. the ordered-list-survives-blank-line quirk).

3. **Platform substitutes for browser APIs:**
   - Leaflet → **osmdroid** (same OpenStreetMap tiles, no API key — keeps parity with the web
     map and avoids a Google Maps key/billing dependency).
   - `navigator.geolocation` → Android `LocationManager` one-shot fix (`location/LocationProvider.kt`),
     runtime-permission gated.
   - `navigator.language` → `LocaleListCompat` device preferences.

4. **App packaging:** a Gradle project, `AndroidManifest.xml`, a **network-security config**
   (Android blocks cleartext HTTP by default since API 28 — the dev backend is plain-HTTP
   loopback, reached at `10.0.2.2:3000` from the emulator), themes, and an adaptive launcher icon.

### What we deliberately do **not** create
- No second AI call path. The app never embeds a model or key; every Claude call still goes
  through `POST /api/chat`. (Same invariant the iOS app holds.)
- No on-device storage of chat/profile (no Room, no SharedPreferences for content) — matches
  the web client's storage-free design. `allowBackup=false`.
- No analytics/tracking SDK. No Google Play Services dependency (FusedLocation avoided so the
  app runs on non-GMS devices; framework `LocationManager` is used instead).

## 3. Architecture mapping (iOS → Android)

The iOS app already proved the shape; Android mirrors it one-to-one.

| Concern | iOS (SwiftUI) | Android (Compose) |
|---|---|---|
| App entry | `AsylumAidApp` | `MainActivity` + `@Composable RootScreen` |
| State (in-memory only) | `AppState: ObservableObject` | `AppViewModel: ViewModel` (`StateFlow`) |
| Wire models | `Models.swift` (Codable) | `model/Models.kt` (`@Serializable`) |
| JSON GETs | `APIClient` (URLSession) | `net/ApiClient.kt` (OkHttp) |
| NDJSON stream | `ChatStream` (`bytes.lines`, `AsyncThrowingStream`) | `net/ChatStream.kt` (`readUtf8Line`, `Flow`) |
| i18n | `LocalizationStore` | `i18n/LocalizationStore.kt` |
| Countries | `Countries` (`Locale`) | `i18n/Countries.kt` (`Locale`) |
| Markdown | `MarkdownRenderer` → `AttributedString` | `md/MarkdownRenderer.kt` → blocks; `MessageBubble` → `AnnotatedString` |
| Theme | `Theme.swift` | `ui/theme/Theme.kt` |
| Screens | `RootView`, `LanguageGate`, `IntakeForm`, `Chat`, `MessageBubble`, `Resources`, `MapHelp` | same names under `ui/` |
| Map | MapKit | osmdroid (OSM) |
| Location | CoreLocation | `LocationManager` |
| Bundled strings | `gen-ios-strings.js` → `Resources/*.json` | `gen-android-strings.js` → `assets/*.json` |
| Dev test w/o IDE | `CoreTests/` (swiftc) | `app/src/test/` (JUnit, `./gradlew test`) |

### Request flow (unchanged from web/iOS)
`IntakeForm` → `AppViewModel.submitIntake` builds a `Profile`, composes the first message via
`buildInitialMessage`, flips to the chat stage, and calls `send` → `ChatStream` POSTs
`{language, profile, messages}` and emits deltas → the assistant bubble grows live → markdown
renders progressively. Error bubbles are display-only and never re-sent (the `messages.wire`
filter), exactly like the web/iOS clients.

## 4. Key fidelity decisions (so behavior matches)

- **Profile JSON shape:** optional fields (`gender`, `civilStatus`, `notes`) are *omitted* when
  empty, not sent as `null` — the web client sends `undefined`. kotlinx.serialization with
  `explicitNulls = false` reproduces this; the server's `validateChatBody` then sanitizes.
- **Markdown safety:** the renderer produces a typed block model (no HTML), links are restricted
  to `http(s)`, and blank lines inside a list do **not** restart numbering — a direct port of the
  web `renderMarkdown` and the iOS `MarkdownRenderer`. Verified by unit test.
- **i18n fallback:** `strings[lang][key] ?? strings["en"][key] ?? key`, with RTL derived from
  `languages.json` (plus a hard-coded `ar`). Same chain as `i18n.t`.
- **Map priority:** osmdroid map is labeled **unverified community data**; the curated
  "Find legal help" directory stays the authoritative path, and the backend's
  NGO-before-lawyer ranking is preserved (it's server-side, so it just works).
- **No new translation service:** the 100-language tables are reused verbatim from the committed
  web assets through `gen-android-strings.js` (pre-resolves the hand-written→machine→English
  merge into one flat JSON), so the app ships without a build-time API key.

## 6. Verification plan

- **No-IDE core test:** `./gradlew test` runs `CoreTest.kt` on the JVM — it loads the bundled
  `assets/*.json`, asserts 100 languages, the i18n fallback chain, RTL flags, the markdown
  quirks (bold stripped, http-only links, ordered list across a blank line), `buildInitialMessage`
  placeholder filling, and the wire-history / optional-field-omission JSON contracts. This is the
  Android analogue of `ios/CoreTests/main.swift`.
- **Contract check:** the same `CoreTest.kt` decodes representative `/api/resources|geocode|places`
  and NDJSON payloads through the `@Serializable` models, so a backend response-shape drift fails a test.
- **Live smoke (emulator):** `npm start` (repo root) → run the app → pick **Arabic** (RTL) → fill
  intake → watch the reply stream → open **Find legal help** → open the **map** → **Use my location**.
- **Backend stays green:** the existing in-process endpoint checks in `CLAUDE.md` are unaffected
  because the server is unchanged.
